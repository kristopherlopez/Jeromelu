# Data Sources

Profiles of every JSON archive in S3 — what fields each upstream feed carries, with type, presence rate, cardinality, and example values across a sample of recent objects. Generated by `scripts/profile_s3_json.py`.

These profiles are the **upstream-side ground truth** for [data-catalogue](../data-catalogue/README.md) lineage stanzas: when you write "`matches.attendance` comes from `nrlcom/match-centre`", the linked profile shows you the actual JSON path (`$.attendance`) with example values (`17476, 12570, 25938`). No more guessing what's in the JSON.

## Reading a profile

Each file has three sections:

1. **Sample objects** — exact S3 keys the profile was built from
2. **Top-level structure** — tree of object/array containers and their immediate children, for one-glance orientation
3. **Field profile** — every leaf path with type, presence (`X/N` of samples that contain it), nullability, distinct value count, and up to 3 example values

The `__len__` rows tell you the typical array size at that path (e.g. `$.awayTeam.players.__len__` cardinality 1, value 19 → always 19 players in a named lineup).

Paths use jsonpath-style notation. `[*]` denotes an array element; per-element values aggregate across all elements in the sample, so `players[*].playerId` cardinality 152 means 152 distinct player IDs across 8 matches × 19 players (with overlap).

## Regenerating

```bash
# One pipeline, default 10 samples from the most-recent objects under the prefix
services/api/.venv/Scripts/python scripts/profile_s3_json.py \
  "miner/nrlcom/match-centre/111/2026/round-07/" \
  --out docs/operations/data-sources/nrlcom/match-centre.md

# Larger sample, spread strategy (evenly across the historical archive)
python scripts/profile_s3_json.py "miner/nrlcom/stats/" --samples 20 --strategy spread

# Defaults: bucket=jeromelu-clean-documents, samples=10, strategy=last
```

When an upstream feed's shape changes (a Miner endpoint-drift test fails, or you add a new field on a partner site), regenerate the affected profile in the same PR. The profile diff is the human-readable record of what changed.

## Sample-prefix choices

Most profiles target a prefix that's deliberately scoped to give a representative sample:

| Profile | Prefix used | Rationale |
|---|---|---|
| nrlcom/match-centre | `…/111/2026/round-07/` | Round 7 was complete with full stats; broader prefix would surface future-game stubs |
| nrlcom/draw | `…/draw/111/2026/` | Latest season's rounds |
| nrlcom/ladder | `…/ladder/` | Broad — only ~12 ladder snapshots total |
| nrlcom/players-roster | `…/players-roster/111/` | All NRL-comp teams (~17 files) |
| nrlcom/stats | `…/stats/111/` | Sparse — 3 season files |
| nrlcom/casualty-ward | `…/casualty-ward/` | Only 1 daily snapshot today |
| supercoach/* | source roots | 1–2 dailies per pipeline; full prefix gets all of them |
| nrlsupercoachstats/stats | `…/stats/2026/` | Latest season's per-round stat dumps |

Re-scope when fixtures backfill more history.
